# 第1章 项目需求分析

## 1.1 案例题目
**题目：基于 PySpark 的校园食堂客流与备餐预测系统**

本系统面向校园食堂运营场景，通过在线点餐积累订单数据，利用 PySpark 进行离线大数据分析，为食堂管理者提供客流统计、高峰时段识别、热门菜品排行和备餐数量预测等数据决策支持。

---

## 1.2 为什么选择 Spark
本项目的核心计算任务是**对食堂订单数据进行离线统计分析**，备选的大数据计算引擎包括 Spark、Flink、MapReduce 和纯 Python 脚本。最终选择 PySpark，理由如下：

### 1.2.1 Spark vs MapReduce

| 维度 | Spark | MapReduce |
|------|-------|-----------|
| 计算模型 | 内存计算 + DAG 调度 | 磁盘中间结果 |
| 处理速度 | 比 MapReduce 快 10-100 倍 | 慢 |
| 编程模型 | RDD/DataFrame/SQL 多种 API | 仅 Map + Reduce |
| 适用场景 | 批处理、迭代计算、交互查询 | 大规模离线批处理 |

本项目需要对订单数据进行分组聚合、排序、时间序列分析等操作，涉及多次数据变换，MapReduce 的 Map-Shuffle-Reduce 模型会导致大量磁盘 I/O，而 Spark 的 DAG 调度可将多个转换合并执行，大幅提升效率。

### 1.2.2 Spark vs Flink

| 维度 | Spark | Flink |
|------|-------|-------|
| 核心定位 | 批处理为主，支持微批流处理 | 流处理为主，批流一体 |
| 时间语义 | 基于事件时间的处理较复杂 | 原生支持事件时间 |
| 成熟度 | 2009年起源，生态完善 | 2014年起源，发展迅速 |
| 社区规模 | 更大，资料丰富 | 快速增长中 |

本项目的分析场景是**离线批处理**（每日凌晨定时分析前一天订单数据），而非实时流处理。食堂订单数据不需要毫秒级实时响应，Spark 的批处理能力完全满足需求且更加成熟稳定。对于课程设计而言，Spark 的学习资料和社区支持更丰富，有利于开发和调试。

### 1.2.3 Spark vs 纯 Python 脚本

| 维度 | PySpark | 纯 Python (Pandas) |
|------|---------|---------------------|
| 数据规模 | TB 级别 | GB 级别（受内存限制） |
| 并行度 | 分布式并行计算 | 单机多核 |
| 扩展性 | 可线性扩展至数百节点 | 无法横向扩展 |
| SQL 支持 | SparkSQL 完整支持 | 需借助 SQLite 等 |

虽然本课程设计的数据集规模在 GB 级别以内，纯 Python 脚本也能处理，但课程本身要求演示 Spark 的应用编程能力，因此选择 PySpark 来构建分析模块。

### 1.2.4 选择 PySpark 的综合原因

1. **批处理匹配**：离线分析场景天然适合 Spark 批处理，不需要 Flink 的实时能力
2. **Python 生态**：PySpark 支持 Python 语言，能与 pandas、matplotlib 等库结合使用
3. **SQL 友好**：SparkSQL 可直接对订单数据执行分组聚合、排序等 SQL 操作，开发效率高
4. **架构清晰**：Spark 作为独立分析层，与 SpringBoot 业务层通过 MySQL 解耦，职责分明
5. **课程要求**：大作业明确要求基于 Spark 开发，PySpark 是最直接的实现方式

## 1.3 功能需求
### 1.3.1 系统总体需求

本系统分为三大子系统：**在线点餐系统**（面向师生用户）、**后台管理系统**（面向食堂管理员）、**大数据分析系统**（PySpark 离线执行）。

### 1.3.2 在线点餐功能（用户端）

| 编号 | 功能 | 描述 |
|------|------|------|
| FR-U-01 | 用户注册 | 用户名 + 密码注册，校验用户名唯一性 |
| FR-U-02 | 用户登录 | JWT token 认证，登录后可点餐 |
| FR-U-03 | 分类浏览 | 按分类（如主食、炒菜、汤品）浏览菜品 |
| FR-U-04 | 菜品搜索 | 支持按菜品名模糊搜索 |
| FR-U-05 | 菜品详情 | 查看菜品图片、描述、价格 |
| FR-U-06 | 提交订单 | 选择菜品和数量，生成订单 |
| FR-U-07 | 查看订单 | 分页查看个人历史订单及明细 |

### 1.3.3 后台管理功能（管理员端）

| 编号 | 功能 | 描述 |
|------|------|------|
| FR-A-01 | 分类管理 | 菜品分类的 CRUD，排序调整 |
| FR-A-02 | 菜品管理 | 菜品的 CRUD，上下架操作 |
| FR-A-03 | 订单管理 | 查看所有订单，修改订单状态 |
| FR-A-04 | 用户管理 | 查看用户列表，启用/禁用用户 |
| FR-A-05 | 数据看板 | 展示分析结果（客流/高峰/销量/预测） |

### 1.3.4 大数据分析功能 (PySpark)

| 编号 | 功能 | 描述 |
|------|------|------|
| FR-P-01 | 客流分析 | 按天统计订单量、销售额、客单价、消费用户数 |
| FR-P-02 | 高峰时段分析 | 按小时统计订单量，识别就餐高峰时段 |
| FR-P-03 | 热门菜品分析 | 统计菜品销量和销售额 TOP10 |
| FR-P-04 | 备餐预测 | 基于历史数据预测未来菜品销量，给出建议备餐量 |

### 1.4.2 用例描述（核心用例）

#### 用例 UC-01：提交订单

| 属性 | 描述 |
|------|------|
| 用例编号 | UC-01 |
| 用例名称 | 提交订单 |
| 参与者 | 普通用户 |
| 前置条件 | 用户已登录，已选择至少一个菜品 |
| 后置条件 | 订单生成，状态为"待支付" |
| 基本流程 | 1. 用户进入菜品浏览页 2. 选择菜品和数量加入购物车 3. 确认订单信息 4. 点击提交 5. 系统生成订单号并保存 6. 返回订单确认信息 |
| 异常流程 | 菜品已下架时提示用户刷新；库存不足时提示并阻止下单 |

#### 用例 UC-02：执行备餐预测（PySpark 分析）

| 属性 | 描述 |
|------|------|
| 用例编号 | UC-02 |
| 用例名称 | 执行备餐预测分析 |
| 参与者 | PySpark 分析系统（定时触发） |
| 前置条件 | MySQL 中存在至少 7 天的历史订单数据 |
| 后置条件 | 预测结果写入 meal_prediction 表，前端可查询展示 |
| 基本流程 | 1. 定时任务触发 main.py 2. SparkSession 初始化 3. 通过 JDBC 读取菜品销量历史数据 4. 计算移动平均 5. 生成预测结果 6. 写回 MySQL meal_prediction 表 7. 关闭 SparkSession |
| 异常流程 | 数据不足时跳过预测并记录日志 |

#### 用例 UC-03：管理员查看数据看板

| 属性 | 描述 |
|------|------|
| 用例编号 | UC-03 |
| 用例名称 | 查看数据分析看板 |
| 参与者 | 管理员 |
| 前置条件 | 管理员已登录，PySpark 已执行过分析 |
| 后置条件 | 展示客流趋势图、高峰时段图、菜品销量排行、备餐预测表 |
| 基本流程 | 1. 管理员进入数据看板 2. 前端调用分析接口 3. 后端查询分析结果表 4. 返回 JSON 数据 5. Echarts 渲染图表 |
